# рҹӨ– Viha WhatsApp AI Sales Assistant вҖ” V2 RAG Agent

> A production-grade conversational AI sales agent that qualifies return gift leads on WhatsApp вҖ” collecting requirements naturally, searching products semantically, and handing off to humans at the right moment.

Built for **Viha Return Gifts**, a real Indian business selling return gifts for weddings, baby showers, corporate events and functions in Chennai.

---

## рҹҺҜ What It Does

Customers message the WhatsApp number asking about return gifts. The AI agent:

1. **Greets** the customer warmly
2. **Collects** all 4 required fields naturally вҖ” quantity, budget, delivery location and timeline
3. **Never reveals prices** until all 4 fields are collected вҖ” filters out casual browsers
4. **Searches semantically** using RAG (pgvector + Jina embeddings) to find best matching products
5. **Sends product images** with name and price in caption вҖ” after requirement summary
6. **Hands off silently** to the human team вҖ” no message to customer, just wife gets alerted
7. **Stays silent** вҖ” bot never interferes after human takes over

The result: wife/staff only speaks to **qualified leads** with full context already collected.

## рҹ”Қ RAG Implementation

V2 adds semantic product search using pgvector and Jina AI embeddings вҖ” the main technical addition over V1.

```
Product Embedding (one-time, run locally):
  embed_products.py вҶ’ Jina API вҶ’ 768-dim vectors вҶ’ product_embeddings table

Search at runtime:
  Customer query вҶ’ Jina API вҶ’ query vector
                вҶ’ pgvector cosine similarity
                вҶ’ top matching products
                вҶ’ filter by budget + quantity
                вҶ’ return ranked results
                вҶ’ SQL fallback if RAG returns empty
```

**Why Jina AI?**
- Free tier: 1M tokens/month
- No local model вҖ” saves ~400MB RAM on Render free tier
- 768-dimension embeddings for high quality semantic matching

**Why RAG over SQL-only?**

V1 SQL search filters by exact budget and quantity вҖ” works well but cannot handle semantic queries like "eco friendly gifts" or "something traditional for south Indian wedding." V2 RAG matches customer intent to product descriptions semantically, returning relevant results even for vague queries.

## рҹ’Ў Key Business Logic

**4 Fields Gate:**
```
Bot NEVER shows products or prices until ALL 4 collected:
  вң“ Quantity    (how many pieces)
  вң“ Budget      (price per piece)
  вң“ Location    (delivery city)
  вң“ Timeline    (when needed)

Casual browsers who don't answer вҶ’ no products shown
Serious buyers who answer all 4  вҶ’ products shown with prices вң…
```

**Natural Collection:**
```
Customer gives 2 fields вҶ’ bot asks for missing 2 together (numbered list)
Customer gives all 4    вҶ’ bot immediately searches and shows products
Never asks one field at a time
```

**Silent Handoff:**
```
After products shown вҶ’ wife alerted with full requirements
Customer says anything вҶ’ bot stays completely silent
No message to customer вҖ” seamless handoff
```

## рҹ’Ў Key Design Decisions

**Why single agent node instead of multi-node pipeline?**
V1 routes every message through a fixed sequence of specialized nodes regardless of context вҖ” greeting вҶ’ intent вҶ’ extraction вҶ’ validation вҶ’ search вҶ’ recommendation. This works well for standard flows but adds latency (multiple LLM calls per turn) and is harder to maintain as business logic grows. A single conversational agent with tools handles the entire conversation in one LLM call, decides freely which tool to use based on context, and is easier to update by modifying the system prompt.

**Why RAG instead of SQL-only search?**
SQL filters work well for exact budget and quantity matching. But customers often describe needs vaguely вҖ” "eco friendly", "something traditional", "good for south Indian wedding." RAG matches these semantic queries to product descriptions, returning relevant results that SQL would miss.

**Why Jina AI instead of local sentence-transformers?**
Local sentence-transformers model uses ~400MB RAM вҖ” exceeds Render free tier 512MB limit causing OOM crashes. Jina API has no local model, keeps RAM at ~150MB, and provides free 1M tokens/month.

**Why gate products behind 4 fields?**
Based on 1.5 years of real business experience вҖ” casual enquirers who don't share requirements rarely convert. Gating product images behind 4 fields ensures wife only follows up with qualified leads who have real buying intent.

**Why two services?**
Node.js handles WhatsApp (Baileys requires Node). Python handles AI (LangChain/LangGraph ecosystem). Clean separation with HTTP between them.

**Why Groq?**
Fast inference (under 3 seconds per response) with Llama 3.3 70B. Free tier sufficient for current volume.

## рҹ”„ Why V2? вҖ” The Switch from V1

V1 worked well in production. The conversation flow, 4 fields collection, silent handoff and admin commands all worked correctly.

**The only limitation was product search.**

V1 used SQL filters вҖ” matches products by exact budget and quantity. This works perfectly for straightforward queries. But as the catalogue grows, customers describe needs vaguely:

- *"Something eco friendly"*
- *"Traditional gifts for Tamil wedding"*
- *"Unique and different under 50 rupees"*

SQL cannot handle these. It either returns nothing or ignores the semantic intent entirely.

**V2 adds RAG вҖ” semantic product search using pgvector and Jina AI embeddings.**

Every product is embedded as a 768-dimension vector. When a customer describes their need, the query is embedded the same way and compared against all products using cosine similarity. Products are ranked by how well they match the customer's intent вҖ” not just their price and quantity.

The single conversational agent is a secondary architectural simplification вҖ” same business logic, cleaner codebase.

**In short:** V1 = price and quantity matching. V2 = intent and meaning matching.
